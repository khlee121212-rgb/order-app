# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 폴스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

# 프런트엔드 UI PRD - 주문하기 화면

## 4. 프런트엔드 UI PRD - 주문하기 화면

### 4.1 화면명
주문하기 화면 (Order Screen)

### 4.2 화면 목적
사용자가 커피 메뉴를 둘러보고, 옵션을 선택해 장바구니에 담은 뒤 주문을 완료할 수 있는 고객용 화면

### 4.3 사용자
- 카페를 방문한 고객 (별도의 로그인/인증 없이 누구나 이용)

## 5. 화면 구성

주문하기 화면은 위에서부터 아래로 다음 세 영역으로 구성된다.

1. 상단 헤더 (Header)
2. 메뉴 목록 (Menu List)
3. 장바구니 (Cart)

```
┌─────────────────────────────────────────────┐
│  COZY                      [주문하기]  관리자  │  ← 헤더
├─────────────────────────────────────────────┤
│  ┌────────┐  ┌────────┐  ┌────────┐          │
│  │ 메뉴1   │  │ 메뉴2   │  │ 메뉴3   │   ...     │  ← 메뉴 목록
│  └────────┘  └────────┘  └────────┘          │
├─────────────────────────────────────────────┤
│  장바구니                                      │  ← 장바구니
│   - 항목 목록 / 총 금액 / 주문하기 버튼          │
└─────────────────────────────────────────────┘
```

## 6. 영역별 상세 요구사항

### 6.1 상단 헤더 (Header)

| 항목 | 설명 |
| --- | --- |
| 로고 | 좌측에 서비스명 `COZY` 텍스트 표시 |
| 내비게이션 | 우측에 `주문하기`, `관리자` 두 개의 메뉴 표시 |
| 활성 상태 | 현재 화면인 `주문하기`가 선택(강조) 상태로 표시 |
| 이동 | `관리자` 클릭 시 관리자 화면으로 이동 |

- 헤더는 화면 상단에 고정되며 모든 영역에서 항상 보인다.

### 6.2 메뉴 목록 (Menu List)

- 카드 형태의 메뉴 아이템을 가로로 나열한다. (와이어프레임 기준 한 줄 3개)
- 화면 너비가 좁아지면 카드가 자연스럽게 줄바꿈되어 배치된다. (반응형)
- 메뉴 데이터는 백엔드 API에서 조회한다.

각 메뉴 카드의 구성 요소:

| 요소 | 설명 |
| --- | --- |
| 메뉴 이미지 | 메뉴 대표 이미지 (이미지가 없을 경우 placeholder 표시) |
| 메뉴 이름 | 예) `아메리카노(ICE)` |
| 가격 | 예) `4,000원` (천 단위 콤마 표기) |
| 메뉴 설명 | 메뉴에 대한 간단한 설명 텍스트 |
| 옵션 선택 | 체크박스 형태의 옵션 목록 (예: `샷 추가 (+500원)`, `시럽 추가 (+0원)`) |
| 담기 버튼 | 선택한 옵션과 함께 해당 메뉴를 장바구니에 담는 버튼 |

옵션 동작:
- 각 옵션은 체크박스로 다중 선택 가능하다.
- 옵션명 옆에 추가 금액을 `(+금액원)` 형태로 표시한다.
- 추가 금액이 0원인 옵션도 `(+0원)`으로 표기한다.

담기 버튼 동작:
- `담기` 클릭 시 현재 카드에서 선택된 옵션과 함께 해당 메뉴 1개가 장바구니에 추가된다.
- 동일한 메뉴 + 동일한 옵션 조합을 다시 담으면 장바구니에서 수량이 증가한다.
- 동일한 메뉴라도 옵션 조합이 다르면 장바구니에 별도 항목으로 추가된다.

### 6.3 장바구니 (Cart)

- 메뉴 목록 하단에 박스 형태로 표시된다.
- 영역 상단에 `장바구니` 제목을 표시한다.

장바구니 항목(라인) 구성:

| 요소 | 설명 |
| --- | --- |
| 메뉴 이름 + 옵션 | 예) `아메리카노(ICE) (샷 추가)` — 선택된 옵션을 함께 표기 |
| 수량 | 예) `X 1`, `X 2` |
| 금액 | 해당 항목의 (단가 + 옵션가) × 수량, 예) `4,500원` |

장바구니 요약/액션:
| 요소 | 설명 |
| --- | --- |
| 총 금액 | 모든 항목의 합계 금액 표시 (예: `총 금액 12,500원`) |
| 주문하기 버튼 | 클릭 시 장바구니 내용으로 주문을 생성(전송) |

상태 처리:
- 장바구니가 비어 있을 때는 안내 문구(예: "담은 메뉴가 없습니다")를 표시하고, `주문하기` 버튼은 비활성화한다.
- 항목이 1개 이상일 때만 주문이 가능하다.

## 7. 주요 사용자 흐름 (User Flow)

1. 사용자가 주문하기 화면에 진입하면 메뉴 목록이 표시된다.
2. 원하는 메뉴 카드에서 옵션을 선택한다.
3. `담기` 버튼을 눌러 장바구니에 추가한다. (필요한 만큼 반복)
4. 장바구니에서 담긴 항목, 수량, 총 금액을 확인한다.
5. `주문하기` 버튼을 눌러 주문을 완료한다.

## 8. 금액 계산 규칙

- 항목 단가 = 메뉴 기본 가격 + 선택된 옵션들의 추가 금액 합
- 항목 금액 = 항목 단가 × 수량
- 총 금액 = 모든 장바구니 항목 금액의 합
- 모든 금액은 천 단위 콤마와 `원` 단위로 표기한다. (예: `12,500원`)

## 9. 예외 / 상태 처리

| 상황 | 처리 |
| --- | --- |
| 메뉴 데이터 로딩 중 | 로딩 표시 |
| 메뉴 조회 실패 | 오류 메시지 표시 |
| 품절 메뉴 | (관리자 재고 연동 시) 담기 버튼 비활성화 또는 품절 표시 |
| 장바구니 비어있음 | 안내 문구 표시 + 주문하기 버튼 비활성화 |
| 주문 전송 실패 | 오류 메시지 표시 후 재시도 가능 |

## 10. 비고

- 본 화면은 학습 목적이므로 사용자 인증 및 결제 기능은 포함하지 않는다.
- 메뉴는 커피 메뉴만 다룬다.
- 디자인 시안은 첨부된 와이어프레임을 기준으로 하며, 세부 스타일(색상, 폰트 등)은 추후 정의한다.

# 프런트엔드 UI PRD - 관리자 화면

## 11. 개요

### 11.1 화면명
관리자 화면 (Admin Screen)

### 11.2 화면 목적
관리자가 주문 현황을 한눈에 파악하고, 메뉴 재고를 관리하며, 들어온 주문의 상태를 변경(제조 진행)할 수 있는 운영용 화면

### 11.3 사용자
- 카페 운영자/관리자 (학습 목적이므로 별도의 로그인/인증 없이 접근)

## 12. 화면 구성

관리자 화면은 위에서부터 아래로 다음 네 영역으로 구성된다.

1. 상단 헤더 (Header)
2. 관리자 대시보드 (Dashboard)
3. 재고 현황 (Inventory)
4. 주문 현황 (Orders)

```
┌─────────────────────────────────────────────┐
│  COZY                      주문하기  [관리자]  │  ← 헤더
├─────────────────────────────────────────────┤
│  관리자 대시보드                                │  ← 대시보드
│   총 주문 / 주문 접수 / 제조 중 / 제조 완료      │
├─────────────────────────────────────────────┤
│  재고 현황                                     │  ← 재고 현황
│   ┌──────┐ ┌──────┐ ┌──────┐                │
│   │메뉴 +/-│ │메뉴 +/-│ │메뉴 +/-│   ...        │
│   └──────┘ └──────┘ └──────┘                │
├─────────────────────────────────────────────┤
│  주문 현황                                     │  ← 주문 현황
│   시각 / 메뉴 x 수량 / 금액 / 상태변경 버튼      │
└─────────────────────────────────────────────┘
```

## 13. 영역별 상세 요구사항

### 13.1 상단 헤더 (Header)

| 항목 | 설명 |
| --- | --- |
| 로고 | 좌측에 서비스명 `COZY` 텍스트 표시 |
| 내비게이션 | 우측에 `주문하기`, `관리자` 두 개의 메뉴 표시 |
| 활성 상태 | 현재 화면인 `관리자`가 선택(강조) 상태로 표시 |
| 이동 | `주문하기` 클릭 시 주문하기 화면으로 이동 |

- 헤더는 주문하기 화면과 동일한 컴포넌트를 공유한다.

### 13.2 관리자 대시보드 (Dashboard)

- 영역 상단에 `관리자 대시보드` 제목을 표시한다.
- 주문 상태별 건수를 요약하여 한 줄로 표시한다.

| 지표 | 설명 |
| --- | --- |
| 총 주문 | 전체 주문 건수 |
| 주문 접수 | 상태가 `주문 접수`인 주문 건수 |
| 제조 중 | 상태가 `제조 중`인 주문 건수 |
| 제조 완료 | 상태가 `제조 완료`인 주문 건수 |

- 각 지표는 주문 현황 데이터와 실시간으로 연동되어, 주문 상태가 바뀌면 카운트도 갱신된다.

### 13.3 재고 현황 (Inventory)

- 영역 상단에 `재고 현황` 제목을 표시한다.
- 메뉴별 카드를 가로로 나열한다. (와이어프레임 기준 한 줄 3개)
- 화면 너비가 좁아지면 카드가 자연스럽게 줄바꿈되어 배치된다. (반응형)
- 재고 데이터는 백엔드 API에서 조회한다.

각 재고 카드의 구성 요소:

| 요소 | 설명 |
| --- | --- |
| 메뉴 이름 | 예) `아메리카노 (ICE)` |
| 재고 수량 | 현재 재고 수량 표시 (예: `10개`) |
| 증가 버튼(`+`) | 클릭 시 해당 메뉴 재고 1 증가 |
| 감소 버튼(`-`) | 클릭 시 해당 메뉴 재고 1 감소 |

재고 조절 동작:
- `+` / `-` 클릭 시 변경된 재고 수량이 백엔드에 반영되고 화면에 즉시 갱신된다.
- 재고는 0 미만으로 내려갈 수 없으며, 0일 때 `-` 버튼은 비활성화한다.
- (주문하기 화면 연동) 재고가 0인 메뉴는 주문하기 화면에서 품절 처리될 수 있다.

### 13.4 주문 현황 (Orders)

- 영역 상단에 `주문 현황` 제목을 표시한다.
- 들어온 주문을 목록(행) 형태로 표시한다. (최신 주문이 위로 오도록 정렬)

각 주문 행의 구성 요소:

| 요소 | 설명 |
| --- | --- |
| 주문 시각 | 주문이 생성된 날짜/시간 (예: `7월 31일 13:00`) |
| 주문 내용 | 메뉴명과 수량 (예: `아메리카노(ICE) x 1`), 옵션이 있으면 함께 표기 |
| 금액 | 해당 주문의 총 금액 (예: `4,000원`) |
| 상태 변경 버튼 | 현재 상태에 따라 다음 상태로 전환하는 버튼 (예: `주문 접수`) |

주문 상태 흐름:
- 주문 상태는 `주문 접수` → `제조 중` → `제조 완료` 순으로 진행된다.
- 상태 변경 버튼을 누르면 다음 단계로 전환되고, 버튼 라벨/대시보드 카운트가 갱신된다.
- (제안) 상태별로 버튼 라벨을 다르게 표시한다.
  - `주문 접수` 상태 → 버튼: "제조 시작" (누르면 제조 중)
  - `제조 중` 상태 → 버튼: "제조 완료" (누르면 제조 완료)
  - `제조 완료` 상태 → 버튼 비활성화 또는 완료 표시

## 14. 주요 사용자 흐름 (User Flow)

1. 관리자가 관리자 화면에 진입하면 대시보드/재고/주문 현황이 표시된다.
2. 대시보드에서 현재 주문 상태별 건수를 확인한다.
3. 재고 현황에서 `+`/`-` 버튼으로 메뉴 재고를 조절한다.
4. 주문 현황에서 새 주문을 확인하고, 상태 변경 버튼으로 제조 진행 상태를 업데이트한다.
5. 상태 변경 시 대시보드 카운트가 자동으로 갱신된다.

## 15. 데이터 갱신 규칙

- 주문 현황과 대시보드 카운트는 동일한 주문 데이터를 기반으로 한다.
- 주문 상태 변경 또는 신규 주문 발생 시 대시보드 카운트가 즉시 반영된다.
- 재고 수량 변경은 백엔드에 저장되고 주문하기 화면의 품절 여부와 연동된다.
- 모든 금액은 천 단위 콤마와 `원` 단위로 표기한다. (예: `4,000원`)

## 16. 예외 / 상태 처리

| 상황 | 처리 |
| --- | --- |
| 데이터 로딩 중 | 로딩 표시 |
| 주문/재고 조회 실패 | 오류 메시지 표시 |
| 주문이 없음 | 주문 현황 영역에 "주문 내역이 없습니다" 안내 표시 |
| 재고 0 | `-` 버튼 비활성화, 주문하기 화면 품절 연동 |
| 상태 변경/재고 변경 실패 | 오류 메시지 표시 후 재시도 가능 |

## 17. 비고

- 본 화면은 학습 목적이므로 사용자 인증 및 결제 기능은 포함하지 않는다.
- 메뉴는 커피 메뉴만 다룬다.
- 디자인 시안은 첨부된 와이어프레임을 기준으로 하며, 세부 스타일(색상, 폰트 등)은 추후 정의한다.
- 와이어프레임에 명시되지 않은 상세 동작(상태 전환 버튼 라벨 등)은 제안 사항이며 확정 시 조정한다.

# 백엔드 PRD

## 18. 백엔드 개요

### 18.1 목적
프런트엔드(주문하기/관리자 화면)에 메뉴·주문·재고 데이터를 제공하고 저장하는 REST API 서버를 구축한다.

### 18.2 기술 스택
- 런타임/프레임워크: Node.js, Express
- 데이터베이스: PostgreSQL
- 통신 방식: REST API (JSON)

### 18.3 기본 사항
- 학습 목적이므로 사용자 인증/결제는 포함하지 않는다.
- 메뉴는 커피 메뉴만 다룬다.
- 금액 단위는 원(KRW), 정수로 저장한다.

## 19. 데이터 모델

### 19.1 개요 및 관계
- `Menus` 1 : N `Options` — 옵션은 특정 메뉴에 연결된다.
- `Orders` 1 : N `OrderItems` — 하나의 주문은 여러 주문 항목(메뉴 줄)을 가진다.
- `OrderItems` 1 : N `OrderItemOptions` — 하나의 주문 항목은 여러 옵션을 가질 수 있다.

> 참고: 사용자가 요청한 핵심 모델은 `Menus`, `Options`, `Orders` 3가지다. 다만 한 주문에 여러 메뉴/옵션이 담기므로, `Orders`의 "주문 내용"은 정규화하여 `OrderItems`(주문 항목)와 `OrderItemOptions`(주문 항목 옵션) 하위 테이블로 저장한다.

### 19.2 Menus (메뉴)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | SERIAL (PK) | 메뉴 고유 ID |
| name | VARCHAR(100) | 커피 이름 |
| description | TEXT | 메뉴 설명 |
| price | INTEGER | 기본 가격(원) |
| image_url | VARCHAR(500) | 메뉴 이미지 URL |
| stock | INTEGER | 재고 수량 (관리자 화면에 표시) |
| created_at | TIMESTAMP | 생성 일시 |

### 19.3 Options (옵션)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | SERIAL (PK) | 옵션 고유 ID |
| menu_id | INTEGER (FK → menus.id) | 연결할 메뉴 |
| name | VARCHAR(100) | 옵션 이름 (예: 샷 추가) |
| price | INTEGER | 옵션 추가 금액(원) |

### 19.4 Orders (주문)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | SERIAL (PK) | 주문 고유 ID |
| created_at | TIMESTAMP | 주문 일시 |
| status | VARCHAR(20) | 주문 상태 (`received` / `making` / `done`) |
| total_amount | INTEGER | 주문 총 금액(원) |

- 상태 값과 의미: `received`(주문 접수, 기본값) → `making`(제조 중) → `done`(완료)

#### 19.4.1 OrderItems (주문 항목 — 주문 내용)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | SERIAL (PK) | 주문 항목 ID |
| order_id | INTEGER (FK → orders.id) | 소속 주문 |
| menu_id | INTEGER (FK → menus.id) | 주문한 메뉴 |
| menu_name | VARCHAR(100) | 주문 시점 메뉴명(스냅샷) |
| quantity | INTEGER | 수량 |
| unit_price | INTEGER | 단가(기본가 + 옵션가 합) |
| amount | INTEGER | 항목 금액(unit_price × quantity) |

#### 19.4.2 OrderItemOptions (주문 항목 옵션)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | SERIAL (PK) | ID |
| order_item_id | INTEGER (FK → order_items.id) | 소속 주문 항목 |
| option_name | VARCHAR(100) | 옵션명(스냅샷) |
| option_price | INTEGER | 옵션 금액(스냅샷) |

> 메뉴명/가격을 주문 시점에 스냅샷으로 저장하여, 이후 메뉴 정보가 변경되어도 과거 주문 내역이 그대로 보존되도록 한다.

## 20. 데이터 스키마 기반 사용자 흐름

1. **메뉴 표시**: `Menus`의 데이터를 조회해 주문하기 화면에 표시한다. 이때 `stock`(재고 수량)은 주문하기 화면이 아니라 관리자 화면에 표시한다.
2. **메뉴 담기**: 사용자가 화면에서 커피 메뉴와 옵션을 선택해 담으면, 선택 정보가 장바구니(프런트엔드 상태)에 표시된다.
3. **주문 저장**: 장바구니에서 `주문하기` 버튼을 클릭하면 주문 정보를 `Orders`에 저장한다. 주문 시간과 주문 내용(메뉴, 수량, 옵션, 금액)을 `OrderItems`/`OrderItemOptions`에 함께 저장한다. 동시에 주문 수량만큼 해당 메뉴의 `stock`을 차감한다.
4. **주문 현황 표시 및 상태 변경**: `Orders`의 데이터를 관리자 화면 `주문 현황`에 표시한다. 주문의 기본 상태는 `주문 접수`이며, 상태 변경 버튼을 누르면 `주문 접수` → `제조 중` → `완료` 순으로 전환된다.

## 21. API 설계

### 21.1 공통 사항
- Base URL: `/api`
- 요청/응답 형식: `application/json`
- 금액은 정수(원) 단위
- 성공 시 적절한 HTTP 상태 코드(200/201) 반환, 실패 시 4xx/5xx와 오류 메시지 반환
- 오류 응답 예시:

```json
{ "error": "재고가 부족합니다.", "code": "OUT_OF_STOCK" }
```

### 21.2 메뉴 목록 조회

- **설명**: 주문하기 화면 진입 시 데이터베이스에서 커피 메뉴 목록(옵션 포함)을 불러온다.
- **요청**: `GET /api/menus`
- **응답 (200)**:

```json
[
  {
    "id": 1,
    "name": "아메리카노(ICE)",
    "description": "깔끔하고 시원한 아이스 아메리카노",
    "price": 4000,
    "imageUrl": "https://.../americano-ice.jpg",
    "stock": 10,
    "options": [
      { "id": 1, "name": "샷 추가", "price": 500 },
      { "id": 2, "name": "시럽 추가", "price": 0 }
    ]
  }
]
```

> `stock`은 응답에 포함하되, 화면 표시 정책상 주문하기 화면에서는 노출하지 않고 관리자 화면에서만 표시한다.

### 21.3 주문 생성

- **설명**: 사용자가 `주문하기` 버튼을 누르면 주문 정보를 데이터베이스에 저장하고, 주문 수량만큼 메뉴 재고를 차감한다.
- **요청**: `POST /api/orders`
- **요청 본문**:

```json
{
  "items": [
    { "menuId": 1, "quantity": 1, "optionIds": [1] },
    { "menuId": 2, "quantity": 2, "optionIds": [] }
  ]
}
```

- **처리 규칙**:
  - 금액(단가/항목 금액/총액)은 클라이언트 값을 신뢰하지 않고 서버가 DB의 메뉴/옵션 가격으로 계산한다.
  - 주문 저장과 재고 차감은 하나의 트랜잭션으로 처리한다.
  - 재고가 주문 수량보다 적으면 주문을 거부한다(`409 Conflict`).
  - 생성된 주문의 초기 상태는 `received`(주문 접수).
- **응답 (201)**:

```json
{
  "id": 10,
  "createdAt": "2026-06-17T13:00:00+09:00",
  "status": "received",
  "totalAmount": 12500,
  "items": [
    {
      "menuId": 1,
      "menuName": "아메리카노(ICE)",
      "quantity": 1,
      "unitPrice": 4500,
      "amount": 4500,
      "options": [{ "name": "샷 추가", "price": 500 }]
    },
    {
      "menuId": 2,
      "menuName": "아메리카노(HOT)",
      "quantity": 2,
      "unitPrice": 4000,
      "amount": 8000,
      "options": []
    }
  ]
}
```

### 21.4 주문 단건 조회

- **설명**: 주문 ID를 전달하면 해당 주문 정보를 반환한다.
- **요청**: `GET /api/orders/:id`
- **응답 (200)**: 21.3의 주문 객체와 동일한 형식
- **응답 (404)**: 해당 ID의 주문이 없을 때

### 21.5 추가로 필요한 API (관리자 화면 연동)

사용자 흐름 4를 완성하기 위해 다음 엔드포인트가 추가로 필요하다.

| 메서드 | 경로 | 설명 |
| --- | --- | --- |
| GET | `/api/orders` | 주문 목록 조회(관리자 주문 현황, 최신순) |
| PATCH | `/api/orders/:id/status` | 주문 상태를 다음 단계로 전환 (`received`→`making`→`done`) |
| PATCH | `/api/menus/:id/stock` | 관리자 재고 증감 (예: `{ "delta": 1 }` 또는 `{ "delta": -1 }`, 재고는 0 미만 불가) |

## 22. 비고

- 재고 차감/복구, 주문 상태 전환 등 데이터 정합성이 중요한 작업은 트랜잭션으로 처리한다.
- 주문 항목의 메뉴명·가격·옵션은 주문 시점 값으로 저장(스냅샷)하여 이력 정확성을 보장한다.
- 초기 메뉴/옵션 데이터는 시드(seed) 스크립트로 적재하는 것을 권장한다.
